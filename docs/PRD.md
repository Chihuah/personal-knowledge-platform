## 1. 產品概述

本產品是一個面向個人使用者的私人知識平台，目標是解決使用者在日常瀏覽社群媒體與網路內容時，雖然持續收錄有價值的文章、貼文、影片與連結，卻無法有效整理、搜尋與再利用的問題。

目前使用者主要透過 LINE 私人對話作為臨時收納空間，將 Facebook、Threads、YouTube 與一般網頁的連結傳送到個人筆記中。然而，隨著內容持續累積，這些資料逐漸變成難以整理、難以查找、難以回顧的資訊倉庫。

因此，本專案希望建立一個低摩擦收錄、AI 自動整理、可搜尋查詢、可瀏覽回顧的個人知識平台，讓使用者能把零散收藏轉化為真正可再利用的知識資產。

---

## 2. 問題定義

### 2.1 現況
使用者平時在社群媒體與網頁上遇到有價值的內容時，通常會先執行「收藏」這個動作，例如：
- 在 Facebook 看到值得參考的貼文後按分享
- 複製 Threads 貼文連結
- 收藏 YouTube 影片連結
- 將網址貼到 LINE 私人對話中

這種做法雖然能快速保存資訊，但只完成了「蒐集」這個步驟，並未真正進入整理、理解、連結與再利用階段。

### 2.2 核心痛點
1. 收錄內容持續累積，形成數位囤積。
2. LINE 對話紀錄不是為知識管理設計，缺乏結構化整理能力。
3. 已收錄內容大多只有網址，缺乏摘要、分類、關鍵字與內容索引。
4. 當使用者日後突然想到某個主題時，很難快速從過去收藏中找回需要的內容。
5. 即使蒐集大量資訊，也難以轉化為可持續利用的知識庫。

### 2.3 問題本質
使用者的真正需求，不是單純「收藏更多連結」，而是希望建立一個系統，能夠支援以下循環：

**看到內容 → 快速收錄 → 自動整理 → 日後找回 → 重新利用**

---

## 3. 產品目標

本產品的主要目標如下：

1. 提供低摩擦的內容收錄入口，延續使用者現有的收藏習慣。
2. 將單純的 URL 收藏轉化為具結構化資訊的知識條目。
3. 透過 AI/LLM 自動完成摘要、關鍵字、分類與內容類型判定。
4. 提供儀表板、列表頁與詳細頁，讓使用者能瀏覽與管理知識庫。
5. 提供關鍵字搜尋與後續可擴充的自然語言查詢能力。
6. 降低資訊囤積造成的混亂感，提升內容找回率與再利用率。

## 5. 目標使用者

### 5.1 主要使用者
本產品的主要使用者為單一個人使用者，具備以下特徵：
- 經常瀏覽 Facebook、Threads、YouTube 與各類網站
- 習慣看到有價值的內容就先收藏
- 覺得自己累積了很多資訊，但之後常常找不到
- 希望透過 AI 幫助自己做第一層整理
- 需要一個比 LINE 私人筆記更適合長期知識管理的平台

### 5.2 使用特徵
- 手機端收錄頻率高
- 收藏入口必須簡單直接
- 查詢與回顧可能在桌面端進行
- 對摘要、分類、搜尋品質有實際需求

## 6. 核心使用情境

### 情境一：快速收藏貼文
當使用者在 Facebook 上看到值得參考的貼文時，透過分享按鈕或複製連結，將網址送入個人知識平台。

### 情境二：平台自動整理
平台收到連結後，自動擷取標題、來源、內容文本，並利用 AI 生成摘要、關鍵字與分類，建立知識條目。

### 情境三：日後查找知識
當使用者未來想到某個主題，例如「AI 教學遊戲化」或「知識管理工具」，可以進入平台輸入關鍵字或 query，快速查到過去曾收錄的相關內容。

### 情境四：瀏覽近期收錄內容
使用者進入首頁儀表板後，可以查看最近新增內容、最新文章、分類概況與待整理項目。

## 7. 使用者旅程

### 7.1 收錄旅程
1. 使用者在社群平台看到有價值內容
2. 執行分享或複製連結
3. 將連結送入私人知識平台

3. 系統回傳相關知識條目
4. 使用者點入條目查看摘要與原始內容
5. 找回過去曾收藏的重要資訊

## 9. 功能需求

### 9.1 收錄功能
- 使用者可透過手動貼上 URL 將內容送入平台。
- 系統應驗證 URL 格式。
- 系統應判斷是否為重複收錄。
- 系統建立初始知識條目，並將處理流程交由背景任務執行。

### 9.2 內容擷取功能
- 系統應辨識來源平台，例如 Facebook、Threads、YouTube 或一般網頁。
- 系統應擷取 metadata，包括標題、描述、縮圖、作者、發布時間（若可取得）。
- 系統應盡可能擷取主要內容文字。
- 若為 YouTube，系統應盡可能取得影片標題、描述與字幕或逐字稿。

### 9.3 AI 分析功能
- 系統應使用 AI/LLM 產生短摘要。
- 系統應自動提取關鍵字。
- 系統應自動將內容分類到預設類別。
- 系統應判定內容類型，如文章、貼文、影片、工具介紹或教學資源。
- 分析結果須寫入資料庫，供後續展示與搜尋。

### 9.4 知識條目管理功能
- 每筆收錄內容應建立為一筆唯一知識條目。
- 條目應至少包含：
  - id
  - source_url
  - source_platform
  - title
  - author
  - published_at
  - captured_at
  - cleaned_content
  - short_summary
  - keywords
  - category
  - content_type
  - thumbnail_url
  - processing_status
  - error_message

### 9.5 搜尋與查詢功能
- 系統應支援基本關鍵字搜尋。
- 搜尋範圍至少包括標題、摘要、關鍵字與內容文本。
- 系統應支援條件篩選，包括來源平台、分類、時間與狀態。
- 搜尋結果至少顯示：標題、摘要、關鍵字、來源平台、收錄時間。

### 9.6 儀表板與瀏覽功能
- 首頁儀表板顯示總收藏數、最近新增數、最新收錄內容、分類分布與狀態分布。
- 列表頁顯示所有條目，並支援排序、搜尋與篩選。
- 詳細頁顯示單筆條目的完整資訊。

## 10. 非功能需求

### 10.1 易用性
系統須維持低摩擦收錄流程，特別是在新增連結時，避免冗長表單與複雜操作。

### 10.2 效能
系統在接收到網址後，應立即回應收錄成功訊息；內容解析與 AI 分析可於背景非同步處理。

### 10.3 可維護性
系統應採模組化設計，將收錄、解析、AI 分析、儲存與搜尋功能分離。

### 10.4 可擴充性
系統須預留未來擴充更多來源平台、語意搜尋、分享入口與知識代理功能的空間。

### 10.5 私密性
由於本平台為私人知識平台，所有資料應以個人私有為前提進行儲存與存取控管。

### 10.6 搜尋品質
系統應盡可能保存可搜尋的文字內容，而非僅保存 URL。

## 11. 成功指標

本產品是否成功，不應只看收藏數量，而應觀察以下指標：

1. 使用者新增一筆內容的操作是否足夠簡單快速。
2. 已收錄內容是否能在日後被有效找回。
3. 每筆收藏是否從單純網址提升為具結構化資訊的知識條目。
4. 系統是否有效降低資訊混亂與囤積焦慮。
5. 使用者是否願意持續把此平台當作主要知識收納入口。

MVP 階段可觀察的初步產品指標：
- URL 收錄成功率
- 解析成功率
- AI 摘要成功率
- 搜尋找回率
- 平均處理時間

## 12. 解決方案概述

本系統採用以下核心流程：

**分享/貼上連結 → API 接收 → 內容擷取 → AI 分析 → 儲存資料庫 → 建立搜尋索引 → 前端展示**

整體系統由以下模組組成：
1. Ingestion Module（收錄模組）
2. Parsing Module（內容解析模組）
3. AI Enrichment Module（AI 分析模組）
4. Knowledge Store（知識儲存層）
5. Search & Query Module（搜尋與查詢模組）
6. Dashboard & UI Module（前端展示模組）
7. Background Job Module（背景工作模組）

## 13. 系統設計方向

### 13.1 架構原則
MVP 階段建議採用：
- 前後端分離
- API 驅動
- 背景任務非同步處理
- 模組化單體架構（modular monolith）

### 13.2 建議技術選型
- 前端：Next.js
- 後端：FastAPI
- 資料庫：PostgreSQL
- 背景任務：Celery + Redis
- AI：OpenAI API
- 搜尋：PostgreSQL Full Text Search

### 13.3 Parser 設計
依據不同來源平台採用不同 parser strategy：
- FacebookParser
- ThreadsParser
- YouTubeParser
- GenericWebParser

### 13.4 狀態機
知識條目狀態建議如下：
- received
- queued
- parsing
- parsed
- analyzing
- ready
- failed

## 15. API 概述

MVP 階段建議提供以下核心 API：

- `POST /api/items`：新增網址
- `GET /api/items`：取得條目列表
- `GET /api/items/{id}`：取得單筆條目
- `GET /api/dashboard`：取得首頁儀表板資料
- `POST /api/items/{id}/reprocess`：重新處理條目

## 21. 風險與注意事項

1. 社群平台內容擷取限制可能導致 Facebook / Threads 正文抓取不穩定。
2. 不同來源的內容品質差異大，parser 需有 fallback 策略。
3. LLM 回傳格式可能不穩定，需實作 JSON 驗證與重試。
4. 若內容過長，需有 truncate 或 chunking 策略。
5. MVP 階段應避免功能範圍膨脹，聚焦核心閉環。

## 22. 結論

本專案的核心不是打造一個單純的「書籤工具」，而是建立一個能將零散社群收藏轉化為結構化知識條目的私人知識平台。產品的價值在於打通「收錄、整理、查詢、回顧、再利用」這條鏈路，讓使用者平時累積的資訊，真正變成可持續使用的知識資產。

MVP 階段應優先完成最小可用閉環：

**貼上 URL → 自動解析 → AI 摘要/分類 → 儲存 → 列表/搜尋/儀表板可用**

在此基礎上，再逐步擴充分享整合、語意搜尋與 AI 知識助理等第二階段功能。
